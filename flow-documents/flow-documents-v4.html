مستند فلو سیستم

این نسخه رفتار end-to-end مربوط به Episodic Memory را به جریان‌های فعلی سیستم اضافه می‌کند.

نسخه v4 تاریخ 2026-05-17 Scope Admin / Memory / Agent Workflows

نمای کلی فلو جدید

Authorized Admin / Agent -> POST /api/admin/v1/Episodes/RecordAsync -> FluentValidation + service invariants -> duplicate check by DeduplicationKey -> save Episode + Tags + References -> later search/recent/important queries reuse normalized metadata

1) ثبت episode

کاربر ادمین یا عامل مجاز یک episode ثبت می‌کند. درخواست شامل نوع، اهمیت، summary، زمان وقوع، tagها و referenceهای پایدار است.

API درگیر: POST /api/admin/v1/Episodes/RecordAsync

جدول‌های درگیر: memory_episode، memory_episode_tag، memory_episode_reference

قواعد مهم:

  • حداقل یکی از details، tag یا reference باید وجود داشته باشد.
  • رویداد آینده پذیرفته نمی‌شود.
  • اگر parent episode داده شود، باید وجود داشته باشد.
  • duplicateهای نزدیک با `DeduplicationKey` رد می‌شوند.

2) مشاهده single episode

برای بررسی کامل یک رویداد، سیستم episode را به‌همراه tagها و referenceهایش برمی‌گرداند.

API درگیر: GET /api/admin/v1/Episodes/GetAsync?id=<guid>

جدول‌های درگیر: memory_episode، memory_episode_tag، memory_episode_reference

3) جستجوی episodeها

جستجو بر اساس متن، نوع، اهمیت، source، environment، بازه زمانی، tag، reference، commit sha و correlation id انجام می‌شود.

API درگیر: POST /api/admin/v1/Episodes/SearchAsync

جدول‌های درگیر: memory_episode، memory_episode_tag، memory_episode_reference

رفتار: نتیجه به‌صورت paging برمی‌گردد و بر اساس OccurredAtUtc DESC مرتب می‌شود.

4) دریافت episodeهای recent

این فلو برای یافتن رویدادهای نزدیک زمانی استفاده می‌شود؛ مثلا قبل از تغییر schema یا deployment جدید.

API درگیر: GET /api/admin/v1/Episodes/RecentAsync

جدول‌های درگیر: memory_episode، memory_episode_tag

5) دریافت episodeهای مهم

این فلو برای decisionها، incidentها و findingهای high/critical مناسب است.

API درگیر: GET /api/admin/v1/Episodes/ImportantAsync

جدول‌های درگیر: memory_episode، memory_episode_tag

6) نقش Graphify در این فلو

  • Graphify source of truth برای روابط ساختاری باقی می‌ماند.
  • episodeها فقط referenceهای پایدار را ذخیره می‌کنند.
  • عامل‌ها ابتدا graph را برای context و سپس episodic memory را برای history/queryable events بررسی می‌کنند.
