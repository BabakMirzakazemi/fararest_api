Program Flow Document v7

This version documents organization-existence validation in menu management so orphan menu/category/item records cannot be created under a non-existent business.

Impacted APIs

API Purpose Impacted Tables
POST /api/admin/v1/MenuManagement/CreateMenuAsync Validates that OrganizationId points to a real registered organization before creating a menu. accounts_organization, menu
POST /api/admin/v1/MenuManagement/CreateCategoryAsync Rejects category creation when the parent menu points to a missing organization/business. accounts_organization, menu, menu_category
POST /api/admin/v1/MenuManagement/UpdateCategoryAsync Rejects category updates when the category belongs to an invalid organization/business reference. accounts_organization, menu_category
POST /api/admin/v1/MenuManagement/CreateItemAsync Rejects item creation when the category chain points to a missing organization/business. accounts_organization, menu_category, menu_item
POST /api/admin/v1/MenuManagement/UpdateItemAsync Rejects item updates when the category belongs to an invalid organization/business reference. accounts_organization, menu_category, menu_item

End-to-End Behavior

  1. Client submits menu/category/item management requests.
  2. The service resolves the owning organization from the request or from the related menu/category.
  3. The service verifies that the organization exists in accounts_organization.
  4. If the organization does not exist, the request fails with a validation error and no menu data is created or updated.
  5. If the organization exists, the normal menu/category/item flow continues.

Design Rule

Organization existence is enforced in the application service layer to keep controllers thin, preserve business integrity, and protect both new requests and legacy orphaned records from further propagation.
