مستند فلو احراز هویت و مدیریت نشست

این نسخه رفتار جدید خروج کاربر، ابطال نشست جاری، و اعتبارسنجی نشست در زمان بررسی JWT را توضیح می‌دهد.

نسخه v2 تاریخ 2026-05-13 Scope Auth / Session / Logout

1) نمای کلی رفتار جدید

Login موفق -> JWT صادر می‌شود -> fingerprint هش‌شده توکن در accounts_user_session ذخیره می‌شود -> هر درخواست دارای JWT -> OnTokenValidated -> چک user active + security stamp -> چک وجود session بر اساس hash توکن -> چک revoked نبودن session -> چک expiry نشست -> Logout -> session جاری revoke می‌شود -> cookie توکن حذف می‌شود

APIهای درگیر: POST /api/v1/Auth/LoginWithPasswordAsync، POST /api/v1/Auth/EmailPasswordLoginAsync، POST /api/v1/Auth/OtpLoginAsync، POST /api/v1/Auth/PhoneOtpLoginAsync، POST /api/v1/Auth/LogoutAsync

جدول‌های درگیر: accounts_user_session، AspNetUsers

2) فلو ورود و ثبت نشست

در ورود موفق، علاوه بر تولید JWT، یک رکورد نشست با SessionSecretHash ساخته می‌شود. این هش از خود access token گرفته می‌شود و خود توکن خام در دیتابیس ذخیره نمی‌شود.

فیلد نشستتوضیح
session_public_idشناسه عمومی نشست برای نمایش و revoke دستی
session_secret_hashهش SHA-256 از access token
issued_at / last_seen_at / expires_atزمان‌های امنیتی نشست
revoked_at / revoke_reasonوضعیت ابطال نشست

APIهای درگیر: endpointهای login فعال

جدول‌های درگیر: accounts_user_session

3) فلو Logout نشست جاری

API جدید POST /api/v1/Auth/LogoutAsync برای کاربر احراز هویت‌شده، نشست جاری را بر اساس bearer token جاری پیدا می‌کند، آن را با دلیل user_logout revoke می‌کند، سپس cookie توکن سایت را حذف می‌کند.

این رفتار باعث می‌شود همان access token از درخواست بعدی دیگر نتواند از لایه اعتبارسنجی JWT عبور کند.

APIهای درگیر: POST /api/v1/Auth/LogoutAsync

جدول‌های درگیر: accounts_user_session

4) فلو مدیریت نشست‌ها

4.1 مشاهده نشست‌ها

GET /api/v1/Auth/MySessionsAsync نشست‌های revoke نشده کاربر را برمی‌گرداند و با IsCurrentSession نشست جاری را مشخص می‌کند.

4.2 revoke یک نشست مشخص

POST /api/v1/Auth/RevokeSessionAsync با sessionPublicId یک نشست مشخص را revoke می‌کند.

4.3 revoke سایر نشست‌ها

POST /api/v1/Auth/RevokeOtherSessionsAsync همه نشست‌های دیگر کاربر به‌جز نشست جاری را revoke می‌کند.

APIهای درگیر: GET /api/v1/Auth/MySessionsAsync، POST /api/v1/Auth/RevokeSessionAsync، POST /api/v1/Auth/RevokeOtherSessionsAsync

جدول‌های درگیر: accounts_user_session

5) فلو اعتبارسنجی JWT بعد از این تغییر

در OnTokenValidated علاوه بر چک‌های قبلی، این کنترل‌ها انجام می‌شود:

  1. raw token موجود باشد.
  2. هش توکن محاسبه شود.
  3. نشست متناظر برای همان user و همان hash پیدا شود.
  4. نشست revoke نشده باشد.
  5. نشست منقضی نشده باشد.

در نتیجه اگر نشست revoke شود، همان access token در اولین request بعدی fail می‌شود؛ حتی اگر زمان انقضای JWT هنوز نرسیده باشد.

APIهای درگیر: همه endpointهای محافظت‌شده با JWT

جدول‌های درگیر: accounts_user_session، AspNetUsers
