1) نمای کلی فلو

User Registers (Email/Mobile) -> Default Role = User -> JWT on successful login -> Frontend calls GET /api/v1/Auth/MyAuthorization -> EffectiveAuthorizationService computes permissions: (RolePermissions UNION PlanPermissions UNION UserGrants) - UserRevokes -> Frontend applies route/menu guards, Backend enforces permissions

نکته مهم: مرجع نهایی امنیت Backend است؛ فرانت صرفا برای UX از permission استفاده می‌کند.

2) ثبت‌نام کاربر و مقادیر پیش‌فرض

2.1 ثبت‌نام با ایمیل

فلو: کاربر با ایمیل ثبت‌نام می‌کند، OTP ایمیلی دریافت می‌کند، سپس تایید می‌کند و فعال می‌شود.

APIهای درگیر:

  • POST /api/v1/Auth/RegisterWithEmailAsync
  • POST /api/v1/Auth/ConfirmEmailRegistrationAsync
  • POST /api/v1/Auth/ResendEmailActivationLinkAsync
  • GET /api/v1/Auth/ConfirmEmailRegistrationByLinkAsync

جداول درگیر: AspNetUsers، ConfirmationCode، AspNetUserRoles

2.2 ثبت‌نام با موبایل

فلو: کاربر درخواست OTP پیامکی می‌دهد و با OTP ثبت‌نام را کامل می‌کند.

APIهای درگیر:

  • POST /api/v1/Auth/StartPhoneRegistrationAsync
  • POST /api/v1/Auth/ResendPhoneRegistrationOtpAsync
  • POST /api/v1/Auth/CompletePhoneRegistrationAsync

جداول درگیر: AspNetUsers، ConfirmationCode، AspNetUserRoles

2.3 کاربر بعد از ثبت‌نام چه Role/Permission/Plan می‌گیرد؟

مولفهوضعیت فعلی سیستمنحوه اعمال
Role پیش‌فرضUserدر سرویس احراز هویت با AddDefaultRoleToUserAsync اعمال می‌شود.
Permission پیش‌فرضمستقیم روی کاربر ثبت نمی‌شود؛ از Role + Plan محاسبه می‌شود.از accounts_role_permission و accounts_plan_permission و ...
Plan پیش‌فرضبرای کاربر جدید endpoint ثبت‌نام فعلاً پلن مستقیم نمی‌سازد.Plan می‌تواند بعداً توسط SuperAdmin از API مدیریت ثبت شود. Seed سیستم برای کاربر seed، پلن ENTERPRISE می‌گذارد.

پاسخ:

{ "userId": "guid", "roles": ["User", "Admin"], "permissions": ["home.view", "menu.home", "accounts.view"], "permissionVersion": "16-char-hash" }

4) منطق دسترسی: Role / Plan / User Grants

سرویس محاسبه: EffectiveAuthorizationService

4.1 ترتیب تقدم

  1. Role permissions
  2. Plan permissions (پلن‌های فعال و معتبر زمانی)
  3. User direct grants
  4. کسر User revokes
Effective = (Role ∪ Plan ∪ Grant) - Revoke

4.2 جداول درگیر

  • accounts_permission
  • accounts_role_permission
  • accounts_plan_permission
  • accounts_user_permission_grant
  • accounts_user_permission_revoke
  • accounts_user_plan_subscription
  • AspNetUsers, AspNetRoles, AspNetUserRoles

4.3 version و cache

permissionVersion از stampهای نقش/پلن/گرنت/ریووک/اشتراک کاربر hash می‌شود. cache با کلید version دار نگهداری می‌شود.

5) تعریف SuperAdmin سیستم

در وضعیت فعلی پروژه، SuperAdmin معمولاً از مسیر seed تعریف می‌شود (کاربر seed + تخصیص Roleهای User/Admin/SuperAdmin).

نقش SuperAdmin از طریق AspNetUserRoles و role name = SuperAdmin اعمال می‌شود.

APIهای مدیریتی ویژه SuperAdmin:

  • POST /api/admin/v1/AuthorizationAdmin/UpsertRolePermissionsAsync
  • POST /api/admin/v1/AuthorizationAdmin/UpsertPlanPermissionsAsync
  • POST /api/admin/v1/AuthorizationAdmin/UpsertUserPermissionGrantAsync
  • POST /api/admin/v1/AuthorizationAdmin/UpsertUserPermissionRevokeAsync
  • POST /api/admin/v1/AuthorizationAdmin/UpsertUserPlanSubscriptionAsync

این APIها هم Role-based (SuperAdmin) و هم Permission-based (admin.authz.manage) محافظت می‌شوند.

6) تعریف و ثبت‌نام کارمندان سازمان مشتری

مدل داده‌ای سازمان و کارکنان در دامنه Accounts موجود است:

  • accounts_organization (سازمان)
  • accounts_membership (عضویت کاربر در سازمان)
  • accounts_membership_groups (گروه‌های عضویت)
  • accounts_membership_permissions (مجوزهای سطح عضویت)

فرآیند عملی ثبت کارمند سازمان به‌صورت معماری:

  1. کاربر (کارمند) با یکی از روش‌های Auth ثبت‌نام/ایجاد می‌شود.
  2. برای او رکورد عضویت در accounts_membership ثبت می‌شود.
  3. گروه و در صورت نیاز permissionهای عضویت به او نسبت داده می‌شود.

نکته: در API فعلی این ریپو endpoint اختصاصی CRUD برای Membership هنوز به‌صورت عمومی در AuthController ارائه نشده و باید در فاز بعدی به‌صورت Admin/Organization API تکمیل شود.

7) سطح دسترسی کارمندان سازمان با چه متدی انجام می‌شود؟

در وضعیت فعلی سیستم دو لایه همزمان می‌توانند دخیل باشند:

  1. لایه سراسری AuthZ جدید: Role + Plan + User Grants/Revoke (خروجی MyAuthorization)
  2. لایه سازمانی Accounts: membership_groups و membership_permissions (دامنه سازمان)

برای enforce در endpointها از attribute زیر استفاده می‌شود:

  • ApiCustomAuthorize (Role based)
  • ApiPermissionAuthorize (Permission based)

نمونه عملی: UsersController.AllPaginatedAsync نیازمند permission accounts.view است.

8) اگر کاربر Role و User Grant دارد ولی پلنش تمام شده باشد چه می‌شود؟

در محاسبه دسترسی موثر، فقط plan subscriptionهای فعال و معتبر زمانی لحاظ می‌شوند. پس با اتمام پلن:

  1. Plan permissions از خروجی حذف می‌شوند.
  2. Role permissions همچنان باقی می‌مانند.
  3. User direct grants همچنان باقی می‌مانند مگر revoke شده باشند.

نتیجه: کاربر ممکن است بخشی از دسترسی را از دست بدهد (بخش plan-based) ولی دسترسی‌های role/grant را حفظ کند.

اگر سیاست کسب‌وکاری بخواهد «بدون پلن هیچ دسترسی ماژولی نداشته باشد»، باید یک gate سراسری policy اضافه شود تا plan active را پیش‌شرط اجباری کند.
